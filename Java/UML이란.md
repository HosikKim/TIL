# UML (Unified Modeling Language)

## 개념

소프트웨어 시스템을 시각적으로 표현하기 위한 표준 모델링 언어

**목적**: 복잡한 시스템을 그림으로 표현하여 이해하기 쉽게 만듦

## UML의 필요성

| 용도 | 설명 |
|------|------|
| 의사소통 | 개발자, 기획자 간 공통 언어 |
| 문서화 | 시스템 구조 명확히 표현 |
| 설계 검증 | 개발 전 문제점 사전 발견 |
| 유지보수 | 레거시 시스템 분석 |

## UML 다이어그램 분류

### 구조 다이어그램 (정적 구조)

| 다이어그램 | 용도 |
|-----------|------|
| 클래스 | 클래스 간 관계 (가장 많이 사용) |
| 객체 | 특정 시점의 객체 상태 |
| 컴포넌트 | 모듈 간 의존성 |
| 배포 | 하드웨어, 서버 구조 |
| 패키지 | 패키지 구조, 네임스페이스 |

### 행위 다이어그램 (동적 동작)

| 다이어그램 | 용도 |
|-----------|------|
| 유스케이스 | 사용자 관점 기능 |
| 시퀀스 | 시간 순서대로 상호작용 |
| 활동 | 업무 흐름, 알고리즘 |
| 상태 | 객체의 상태 변화 |
| 커뮤니케이션 | 객체 간 메시지 교환 |

## 주요 다이어그램 상세

### 1. 클래스 다이어그램

**관계 유형**:
- 일반화 (상속): is-a, 빈 삼각형
- 실체화 (구현): 인터페이스, 점선
- 의존: 일시적, 점선 화살표
- 연관: 참조 관계, 실선
- 집합: 약한 전체-부분, 빈 마름모
- 합성: 강한 전체-부분, 채운 마름모

**다중성**:
- 1: 정확히 1개
- 0..1: 선택적
- 1..*: 1개 이상
- *: 무제한

### 2. 유스케이스 다이어그램

**구성**:
- 액터: 사용자, 외부 시스템
- 유스케이스: 시스템 기능 (타원)

**관계**:
- Include: 필수 포함
- Extend: 선택적 확장

### 3. 시퀀스 다이어그램

**구성**:
- 생명선: 객체 존재 시간
- 활성화 박스: 활동 중
- 메시지: 실선(동기), 열린 화살표(비동기)

**활용**:
- API 호출 흐름
- 이벤트 처리 순서

## UML의 장점

- **표준화**: 국제 표준, 누구나 이해 가능
- **가시성**: 복잡한 구조 단순화
- **의사소통**: 팀 간 공통 언어
- **설계 검증**: 문제점 조기 발견
- **유지보수**: 시스템 이해 빠름

## 실무 활용

| 단계 | 활용 |
|------|------|
| 초기 | 요구사항 분석, 유스케이스 |
| 개발 | 클래스 설계, API 플로우 |
| 유지보수 | 코드 분석, 리팩토링 |
| 문서화 | 인수인계, 교육 자료 |

## 주의사항

- **과도한 사용 지양**: 필요한 부분만
- **최신 유지**: 코드와 동기화
- **적절한 상세도**: 목적에 맞게

## 정리

**UML 사용 이유**:
- 시각화로 이해도 향상
- 표준화된 의사소통
- 효율적인 설계

**효과적인 사용**:
- 목적에 맞는 다이어그램 선택
- 필요한 만큼만 작성
