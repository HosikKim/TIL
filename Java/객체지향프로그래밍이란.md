# 객체지향 프로그래밍 (OOP)

## 개념

Object-Oriented Programming의 약자로, 프로그램을 객체들의 모임으로 구성하는 프로그래밍 패러다임

**핵심 아이디어**: 부품(객체)을 먼저 만들고 조립하여 완성품(프로그램) 제작

## 객체(Object)란?

**정의**: 속성과 동작을 가지는 모든 것

**분류**:
- 물리적 객체: 자동차, 책, 사람, 컴퓨터
- 추상적 객체: 회사, 주문, 계좌, 날짜

**구성**:
- 필드(Field): 객체의 속성, 상태 (데이터)
- 메서드(Method): 객체의 동작, 기능 (행위)

**예시**:
- 자동차 객체
  - 필드: 색상, 속도, 배기량
  - 메서드: 시동걸기(), 가속하기(), 브레이크()

## 클래스와 객체

### 클래스 (Class)

**정의**: 객체를 만들기 위한 설계도, 틀

**특징**:
- 객체의 구조와 동작 정의
- 메모리에 직접 할당되지 않음
- 논리적 개념

**비유**: 붕어빵 틀

### 객체 (Object)

**정의**: 클래스로부터 생성된 실체

**특징**:
- 메모리에 실제 할당됨
- 클래스의 인스턴스
- 실체화된 데이터

**비유**: 틀에서 만들어진 붕어빵

### 인스턴스화

**과정**: 클래스 → 객체 생성

**의미**:
- 설계도를 바탕으로 제품 생산
- 하나의 클래스에서 여러 객체 생성 가능
- 각 객체는 독립적

## OOP의 4대 특징

### 1. 추상화 (Abstraction)

**개념**: 공통적인 속성과 기능만 추출

**목적**:
- 복잡성 감소
- 핵심 개념 도출
- 불필요한 세부사항 제거


### 2. 캡슐화 (Encapsulation)

**개념**: 데이터와 메서드를 하나로 묶고 외부 접근 제한

**목적**:
- 데이터 보호
- 내부 구현 숨김
- 인터페이스만 공개

### 3. 상속 (Inheritance)

**개념**: 기존 클래스의 속성과 메서드를 물려받음

**목적**:
- 코드 재사용
- 계층 구조 표현
- 확장 용이

**관계**: is-a 관계 (개는 동물이다)

### 4. 다형성 (Polymorphism)

**개념**: 같은 메서드가 다른 동작 수행

**목적**:
- 유연한 설계
- 확장 가능한 구조
- 코드 간결화

**효과**: 하나의 타입으로 여러 객체 처리

## 객체 간의 관계

### 1. 집합 관계 (Has-a)

**의미**: 완성품과 부품

**특징**:
- 전체-부분 관계
- 독립적 존재 가능

**예시**:
- 자동차 - 엔진
- 컴퓨터 - CPU

### 2. 사용 관계 (Use)

**의미**: 객체가 다른 객체를 사용

**특징**:
- 일시적 관계
- 메서드 파라미터
- 지역 변수

**예시**:
- 사람 - 자동차 (운전)
- 학생 - 도서관 (이용)

### 3. 상속 관계 (Is-a)

**의미**: 일반화와 특수화

**특징**:
- 계층 구조
- 공통 특성 상속
- 확장 관계

**예시**:
- 동물 - 개, 고양이
- 도형 - 원, 사각형

## OOP vs 절차지향 프로그래밍

### 절차지향 (Procedural)

**특징**:
- 순차적 처리
- 함수 중심
- 데이터와 함수 분리

**장점**:
- 빠른 실행 속도
- 간단한 구조
- 직관적 흐름

**단점**:
- 유지보수 어려움
- 코드 재사용 한계
- 대규모 프로젝트 부적합

**적합한 경우**:
- 작은 프로그램
- 성능 중요
- 단순한 로직

### 객체지향 (Object-Oriented)

**특징**:
- 객체 중심
- 데이터와 메서드 통합
- 추상화, 캡슐화, 상속, 다형성

**장점**:
- 재사용성
- 유지보수성
- 확장성
- 협업 용이

**단점**:
- 느린 속도
- 설계 복잡
- 학습 시간

**적합한 경우**:
- 대규모 프로젝트
- 장기 유지보수
- 팀 협업

## 비교표

| 구분 | 절차지향 | 객체지향 |
|------|---------|---------|
| 중심 | 함수 | 객체 |
| 데이터 | 전역 공유 | 객체 내 캡슐화 |
| 재사용 | 함수 단위 | 클래스 단위 |
| 확장 | 어려움 | 쉬움 |
| 유지보수 | 어려움 | 쉬움 |
| 속도 | 빠름 | 상대적 느림 |
| 예시 | C | Java, C++, Python |

## OOP 설계 원칙 (SOLID)

```
S - Single Responsibility
클래스는 하나의 책임만

O - Open/Closed
확장에 열려있고 수정에 닫혀있어야

L - Liskov Substitution
부모를 자식으로 치환 가능

I - Interface Segregation
인터페이스를 작게 분리

D - Dependency Inversion
추상화에 의존
```
