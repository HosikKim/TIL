# 객체지향 프로그래밍 (OOP)

## 개념

Object-Oriented Programming의 약자로, 프로그램을 객체들의 모임으로 구성하는 프로그래밍 패러다임

**핵심**: 부품(객체)을 먼저 만들고 조립하여 완성품(프로그램) 제작

## 객체와 클래스

### 객체 (Object)

**정의**: 속성과 동작을 가지는 모든 것

**구성**:
- 필드: 객체의 상태 (데이터)
- 메서드: 객체의 동작 (기능)

**예시**: 자동차 - 색상, 속도(필드) / 시동걸기(), 가속하기()(메서드)

### 클래스 (Class)

**정의**: 객체를 만들기 위한 설계도

**비유**: 붕어빵 틀(클래스) → 붕어빵(객체)

**특징**: 하나의 클래스에서 여러 객체 생성 가능

## OOP의 4대 특징

| 특징 | 설명 | 목적 |
|------|------|------|
| 추상화 | 공통 속성과 기능만 추출 | 복잡성 감소 |
| 캡슐화 | 데이터와 메서드를 묶고 외부 접근 제한 | 데이터 보호 |
| 상속 | 기존 클래스의 특성을 물려받음 | 코드 재사용 |
| 다형성 | 같은 메서드가 다른 동작 수행 | 유연한 설계 |

## 객체 간의 관계

### Has-a (집합)
- 완성품과 부품
- 예: 자동차 - 엔진

### Use (사용)
- 일시적 관계
- 예: 사람 - 자동차(운전)

### Is-a (상속)
- 일반화와 특수화
- 예: 동물 - 개, 고양이

## 절차지향 vs 객체지향

| 구분 | 절차지향 | 객체지향 |
|------|---------|---------|
| 중심 | 함수 | 객체 |
| 데이터 | 전역 공유 | 캡슐화 |
| 재사용 | 함수 단위 | 클래스 단위 |
| 확장 | 어려움 | 쉬움 |
| 유지보수 | 어려움 | 쉬움 |
| 속도 | 빠름 | 상대적 느림 |
| 적합 | 작은 프로그램 | 대규모 프로젝트 |
| 예시 | C | Java, Python |

## SOLID 원칙

- **S**: 단일 책임 - 클래스는 하나의 책임만
- **O**: 개방/폐쇄 - 확장 가능, 수정 불가
- **L**: 리스코프 치환 - 부모를 자식으로 치환 가능
- **I**: 인터페이스 분리 - 작게 분리
- **D**: 의존성 역전 - 추상화에 의존

## 정리

**OOP 사용 이유**:
- 코드 재사용성
- 유지보수 용이
- 확장성
- 현실 세계 모델링

**적합한 경우**:
- 대규모 프로젝트
- 팀 협업
- 장기 유지보수
